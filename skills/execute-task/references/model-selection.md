# 模型选择 — 派发 subagent 时按角色与复杂度定档

> 用途：阶段 2 派发执行 / review / fix subagent，阶段 3 整体验收时，据此选模型档位。
> 目标：用够用的最省档位控成本、提速度；但**轮次数比单价更影响总成本**——档位选错反而更贵。

## 一 · 铁律

- **派发时必须显式指定模型**——不写等于让 subagent 继承主 agent 当前档位（通常是最贵最强档），
  悄悄废掉分档的意义。
- 三档：**cheap（便宜档）/ standard（标准档）/ most-capable（最强档）**——具体对应平台上哪个模型，
  由当前平台与账号实际可用模型决定，不在此写死。平台参数映射见 [platform-agents.md](platform-agents.md)。

## 二 · 按角色定档

| 派发对象 | 默认档位依据 | 升档信号 |
|---|---|---|
| 执行 subagent | 简报已给验收标准 + 涉及文件、多数是机械性实现 → **cheap** | 简报只有文字描述、需要自己设计接口/跨文件协调 → **standard**；需要架构判断或读懂大范围既有代码 → **most-capable**（少见——设计本该已在 make-design 锁定，出现即提示上游可能漏了决策，别把它当常态） |
| review subagent | 按 diff 规模 / 复杂度 / 风险，跟执行同等判断力档位——机械小改动 → **cheap/standard** | 并发、状态机、安全边界等细微改动 → **most-capable** |
| fix subagent | 通常与被修任务的执行档位一致，或降一档（修复范围比原实现窄） | 复审仍不过、问题涉及原设计 → 停下退回上游，不是无限升档硬修 |
| 阶段 3 整体验收（whole-branch review） | 固定 **most-capable**——不用 session 默认档位，这是全链路唯一的架构级判断点 | — |

## 三 · 复杂度信号（执行 subagent，判断 cheap / standard 的主依据）

- 触及 1-2 个文件 + 简报里验收标准 / 测试用例已给全 → **cheap**（多数任务属于此类，因为 split-task 已经拆细拆清）。
- 简报只有文字描述、没有可直接照抄的实现细节、需要执行方自己拼装多文件 → **standard**。
- 简报本身就是可直接转录的完整代码 + 单文件机械修复 → **cheap**（转录 + 补测试，不需要判断力）。

## 四 · 轮次数比单价更重要

- 便宜档在多步任务上经常要 2-3 倍轮次，折算下来不一定省——**只有单文件机械修复、或简报里已给出完整代码的"转录类"实现任务才用 cheap**。
- 按文字描述实现（没有现成代码可抄）的执行者、以及所有 review 者，**至少用 standard 档兜底**。

## 派发前自查

- 三类派发都**显式指定了模型与必要的推理档位**（Claude Code 用 Agent `model`；Codex 用 agent 配置或派发时支持的 model / reasoning 参数），没有留空任其继承主 agent？
- 执行 / review 的档位判断依据是「简报完整度 + 涉及文件数」，不是拍脑袋？
- 阶段 3 整体验收固定用了 most-capable，没有沿用 session 默认档位？
