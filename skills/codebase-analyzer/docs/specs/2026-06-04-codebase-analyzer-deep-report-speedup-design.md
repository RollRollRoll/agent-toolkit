# Codebase Analyzer 深度报告提速：按章节并行起草设计

> **日期**：2026-06-04
> **状态**：设计待确认
> **产出方式**：brainstorming 协作
> **关联**：在 [2026-06-04-codebase-analyzer-enhancements-design.md](2026-06-04-codebase-analyzer-enhancements-design.md) 之后的又一次增强；复用其"路线 A：扩展现有文件、不新建"决策

---

## 1. 背景与目标

### 1.1 痛点

深度报告档（`template-deep.md`，10 章 + 附录）当前的产出流程，在**证据采集之后**有一条**串行长尾**：主 agent 一个人（1）亲自串联端到端流程，（2）从零写满 10 章。即便大项目的证据采集已能按模块并行（`parallel-strategy.md`），这条"串联 + 写满"长尾仍是单 agent 串行。

体验上有两重痛：

- **真实墙钟长**：单 agent 串行写满 10 章耗时久。
- **没有进度感**：中间长时间无输出，最后突然甩出一大篇。

### 1.2 目标

持久改进 skill，让**深度档 + 大项目**的报告产出**同时**压低这两重痛——既缩短墙钟，又给出连续进度感——且不牺牲 skill 的核心价值（结论有源码证据、不猜、连贯气脉）。

### 1.3 核心思路

把项目**已在第 8 章 g 类采用并验证的"派子 agent 画像 + 主 agent 画装配图"模式，泛化到整章报告**：描述性章节拆给子 agent 并行起草，气脉 / 全局视角章节主 agent 亲自写，并叠加分阶段流式输出。

### 1.4 关键权衡（明示）

- **拿 token 换墙钟**：子 agent 各读一遍源码起草，总 token 上升。用户诉求是少等而非省钱，方向一致。
- **提速成立的前提假设**：主 agent "读 + 聚合 7 章" 比 "从零写 7 章" 快。靠给 drafter 传证据摘要 + 严格模板 + 硬约束，把聚合成本压到最低。
- **一致性风险**：并行起草可能术语不一 / 跨章冲突。缓解：气脉骨架（概览 / 5 / 9 / 10）留主 agent；主 agent 做聚合 + 跨章一致性核对（见 §7）。

## 3. 章节分工（定稿）

### 3.1 🔒 主 agent 亲自写（4 块）

| 章节 | 为什么必须主 agent |
|---|---|
| 一页纸概览（含 ASCII 架构图） | 需全局视角 |
| 第 5 章 端到端代表性流程 | 气脉所在，`parallel-strategy` 已明确"端到端绝不切分" |
| 第 9 章 综合评价 / 亮点 | 基于其他章节事实，须最后写 |
| 第 10 章 未搞清楚 | 汇总各 drafter 上报的盲区 |

### 3.2 🔓 子 agent 并行起草（7 章）

第 1 章定位功能、第 2 章整体架构、第 3 章技术栈、第 4 章代码结构、第 6 章配置运行、第 7 章工程质量、第 8 章专项画像 / 场景表。

### 3.3 边界：第 8 章与现有「按扩展条目并行」的衔接

- 第 8 章一般作为 🔓 交给 chapter-drafter 起草。
- 但当第 8 章命中 **g 类且扩展条目较多**（会触发现有「按扩展条目并行」）时，第 8 章改由**主 agent 直接编排**那条已有路径（派画像子 agent + 自己画装配图），**不再套一层 chapter-drafter**，避免两个并行机制嵌套。
- 无论哪种方式，**g 类装配图始终主 agent 自己画**（与现有规则一致）。

## 4. 编排时间线

```
① 采集证据
   （大项目沿用「按模块并行采集」；中等项目主 agent 自己采）
        ↓
② 主 agent 立刻输出「一页纸概览初版 + 目录地图」→ 进度#1，主动停一下
        ↓
③ 并行（提速核心）：
   ├─ 主 agent：亲自追第 5 章端到端流程
   └─ 子 agents：并行起草 🔓 那 7 章（各返回成稿章节）
        ↓
④ drafter 章节陆续返回 → 主 agent 逐章聚合、统一术语、消冲突、跨章核对
   每完成一批 → 推送用户（进度 #2…n）
        ↓
⑤ 主 agent 收尾：写第 9 章综合评价 / 亮点、第 10 章未搞清楚，
   校正一页纸概览的架构图
        ↓
⑥ 自检（现有第 5 步）→ 问保存（现有第 6 步）
```

- **提速来自**：7 章并行起草 + 主 agent 追端到端与 drafter 起草重叠。
- **进度感来自**：概览先出 + 边就绪边推。
- **批次**：drafter 派发沿用现有"一批不超过 5~6 个、同一条消息内发出才真并行"。7 章可分两批。

## 5. 启用门槛与退化

### 5.1 何时启用

同时满足：

- 输出深度 = **深度档**；
- 子 agent 工具**可用**；
- 项目**够大**（沿用 `parallel-strategy` 现有阈值：源文件 >500 / 顶层模块 >10 / 或预计报告撑满单次输出）。

不满足时不要为并行而并行——中小项目主 agent 串行写更连贯、反而更快。

### 5.2 退化

- 无子 agent 工具 / 项目偏小 → 主 agent 串行写，**保留流式**（概览先出 + 边写边推），报告注明"未启用章节并行"。
- 某章 drafter 失败或质量明显不足 → 主 agent 自己补写该章。

## 6. chapter-drafter 派活模板（新的第三种派活）

与现有「模块采集」「扩展条目画像」两种派活的关键区别：**它返回成稿章节，不是简报**。

```
你是项目调研助手。请起草调研报告的【第 X 章：{{章标题}}】，
直接返回该章的中文 Markdown 成稿（主 agent 会聚合，不要写别的章）。

【项目根目录】{{绝对路径}}
【证据摘要（导航用，不是全部真相）】
  - 一句话定义 / 技术栈速记
  - 目录地图 + 顶层模块清单（带路径）
  - 入口位置、依赖清单要点
【本章骨架】{{从 template-deep 摘出的该章子标题结构}}

【产出要求】
  - 按本章骨架的子标题逐项填，产出成稿（不是简报）
  - 证据摘要不足处，回到源码核实并深化——不准只转写摘要
  - 每个论断带 `path:line`；不凭名字 / README 猜
  - 区分「不适用（原因）」与「未搞清楚（原因）」
  - 中文；篇幅按深度档该章应有的详尽度，不硬撑不敷衍
【必填】本章「未搞清楚」清单（汇给主 agent 第 10 章）
```

- **共用证据摘要**：同一份摘要发给所有 drafter（定义 / 技术栈 / 目录地图 / 模块清单 / 入口 / 依赖要点），省得各自重新发现项目结构；具体源码核实各 drafter 自做。
- **不准只转写摘要**：硬约束要求每个论断带 `path:line`，drafter 必须回源码核实深化，否则等于没有源码证据。

## 7. 聚合纪律

沿用 `parallel-strategy` 现有聚合 5 条（统一术语 / 消冲突 / 归位 / 补端到端 / 合并未搞清楚），**新增 1 条**：

- **跨章一致性核对**：第 2 章架构的模块名 / 职责须与第 4 章代码结构、第 8 章专项一致；第 3 章技术栈不得与第 6 章配置打架。主 agent 读每章时核对，冲突自己回源码定夺，不留给读者。

---

## 8. 流式节奏

沿用现有"大项目分阶段输出节奏"：

- 概览 + 目录地图先出 → **主动停一下**，告知下一步、给用户调整关注点 / 排除章节的机会（即现有阶段 1 checkpoint）。
- 之后每完成一批章节同步一次（不必每章都停，免打断）。
- 最后聚合完整版 → 自检 → 问保存。

## 10. 验收标准

1. `parallel-strategy.md` 有「按报告章节并行起草」节，覆盖：门槛 / 章节分工 / 时间线 / 派活模板 / 聚合 / 流式 / 退化。
2. `SKILL.md` 第 4 步、第 5 步深度档自检、大项目策略、相关参考四处均已串到。
3. `template-deep.md` 顶部有一句并行起草指引，章节骨架未动。
4. 退化路径明确：无工具 / 太小 → 串行 + 流式 + 注明"未启用章节并行"；单章失败 → 主 agent 补写。
5. 正交：仅深度档启用；标准 / 快速档不受影响。
6. 硬约束（`path:line` / 不凭名字猜 / 区分不适用·未搞清楚）写进 drafter 派活模板并继续成立。
7. 第 8 章与「按扩展条目并行」的衔接边界写清（§3.3），两个并行机制不嵌套。

## 11. 不在本次范围（Non-goals）

- 不新建 references 文件（沿用路线 A）。
- 不改动 quick / standard 档的产出方式（章节并行仅深度档）。
- 不改深度档的章节结构 / 数量（10 章不变，只改"谁来写、怎么推送"）。
- 不引入新的输出深度档位。
- 不把端到端流程（第 5 章）拆给子 agent（铁律不变）。
- 不追求省 token（本次明确以墙钟 + 进度感为目标，token 上升可接受）。
