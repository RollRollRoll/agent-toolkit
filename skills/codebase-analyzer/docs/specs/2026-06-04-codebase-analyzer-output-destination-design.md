# Codebase Analyzer 输出目的地选择：终端 / 文件 / 两者

> **日期**：2026-06-04
> **状态**：设计待确认
> **产出方式**：brainstorming 协作
> **关联**：继 [2026-06-04-codebase-analyzer-enhancements-design.md](2026-06-04-codebase-analyzer-enhancements-design.md) 与 [2026-06-04-codebase-analyzer-deep-report-speedup-design.md](2026-06-04-codebase-analyzer-deep-report-speedup-design.md) 之后的又一次增强；沿用"路线 A：扩展现有文件、不新建"

## 1. 背景与目标

### 1.1 痛点

现有 `SKILL.md`「第 6 步：询问是否保存报告」的流程是：**报告无条件先在对话 / 终端里全文输出**，自检通过后**再**问用户"要不要存成文件"。

也就是说整篇报告必然先在终端刷一遍。深度报告档（10 章 + 附录）尤其严重：

- **刷屏**：长报告全文糊在终端，用户被淹没，难快速定位。
- **体验割裂**：与"边出边推、忌长静默后甩大篇"的偏好冲突——全文一次性甩出正是"甩大篇"。
- **没有"只要文件"的出口**：想直接拿一份文件、终端别刷的诉求，现有流程无法表达。

### 1.2 目标

把"选择输出到哪里"从**事后问保存**前置为**事中问目的地**：调研完成、报告在上下文成型、即将输出那一刻，让用户选 **终端 / 文件 / 两者**，从而支持"不在终端刷整份报告"。**只打扰一次**，取代旧的事后问保存。

### 1.3 关键权衡（明示）

- **报告全文必然先在主 agent 上下文里生成**（否则无从写盘）。"选文件"省的不是生成，而是"不把全文作为对话消息贴出来"。
- **"文件"档终端不能完全静默**（违背进度可见偏好），故保留"路径 + 一页纸概览"让用户扫到大局。
- **三档统一都问**：快速概览半屏写文件略重，但用户要规则一致、无例外，接受这点轻微冗余。

## 2. 核心设计决策

| 决策 | 结论 |
|---|---|
| 落地路线 | 扩展现有文件，不新建（沿用路线 A） |
| 询问时机 | 调研完成、自检通过、即将输出时（事中），取代旧"事后问保存" |
| 适用档位 | **三档统一都问**（快速 / 标准 / 深度） |
| 选项集 | **三选**：终端 / 文件 / 两者 |
| "文件"档终端留存 | 路径 + 一页纸概览 |
| 打扰次数 | 只问一次，选完不二次追问存盘 |

**为什么是三选不是二选**：旧流程里选"终端"的人事后还能存盘；新流程取代了"事后问保存"，若只给"终端 / 文件"二选，"想终端看全文、也想存一份"就没法表达了。加"两者"才不丢旧能力，且成本几乎为零。

## 3. 新「第 6 步：询问输出目的地」（定稿文案）

> 替换 `SKILL.md` 现「第 6 步：询问是否保存报告」全文。

报告在对话中产出、且第 5 步自检通过后，**先不要把全文贴出来**，而是问用户希望报告输出到哪里。默认**不自动写盘**。

- **询问方式**：当前环境提供交互式选择工具（如 Claude Code 的 AskUserQuestion）时用它给出下面三个选项；不可用就用纯文本询问，让用户回一个。
- **大项目分阶段产出时**：等全部模块 / 章节聚合、形成完整报告后再问目的地（见 §4）。

**三个选项的精确语义**：

| 选项 | 终端里 | 文件 |
|---|---|---|
| **终端** | 报告全文贴在对话里 | 不写 |
| **文件** | 仅 `✅ 已写入 <路径>` + **一页纸概览** | 写入全文 |
| **两者** | 报告全文贴在对话里 | 写入全文 |

**一页纸概览的来源**（"文件"档终端要留的那段）：

- 标准 / 深度档：直接摘报告里现成的 `## 一页纸概览` 章节。
- 快速概览档：没有该章节，摘它的"一句话定义 + 核心价值"两三行。

**落盘细节**（选"文件"或"两者"时，复用现状）：

- **默认位置**：被调研项目的根目录。
- **默认文件名**：`<项目名>-调研报告-<YYYYMMDD>.md`——`<项目名>` 取被调研目录名，`<YYYYMMDD>` 取当天日期，例如 `codebase-analyzer-调研报告-20260604.md`。
- 用户若指定了自己的路径 / 文件名，以用户的为准。
- 写盘前若发现**同名文件已存在**，先提示用户确认覆盖还是改名，不要静默覆盖。
- 写入内容就是这份报告原文，不为落盘重新精简或扩写。

## 4. 大项目流式协调

大项目分阶段流式输出（`parallel-strategy.md`）与"事中问目的地"的衔接：

- **流式过程只向终端推进度 + 概要**：全局概览、目录地图、"第 X 章已就绪"这类进度信号与概要。
- **逐章成稿不贴终端**：各 drafter 返回的成稿章节在主 agent 上下文里聚合，不逐章糊到终端。
- **聚合完成、形成完整报告后才问目的地**，最终完整报告按所选目的地落定（选文件 = 写盘 + 终端一页纸概览）。

效果：**进度感**（过程有概要 + 进度）与**不刷屏**（成稿全文只在最终按目的地落定）两者兼得。
