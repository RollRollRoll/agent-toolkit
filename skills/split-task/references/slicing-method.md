# 切片与定序方法 — 垂直切片、分级、依赖与并行、覆盖核对

> 用途：`split-task` 招牌机制的操作细则。阶段 1 切片 / 定序、阶段 2 起草任务时用。
>
> **和拆分探针的分工**（两层，别混）：
> - **拆分探针（覆盖面）**：有**哪些**易漏维度要拆任务？防"整类工作被漏"。
> - **本文（任务怎么切、怎么排）**：选定要做的工作，怎么切成可验收的片、定大小、排依赖、对齐 design + spec。

## 一 · 垂直切片：怎么切任务

**垂直切片 = 每个任务穿透实现它所需的各层，交付一条窄而完整、可独立验收的路径**，
而不是按技术层横切（"先做完所有 DB，再做完所有 API，再做完所有 UI"）。

- **好的切片（垂直）**：「实验清单展示」——从查询到模板渲染一条龙，做完就能开页面看到结果。
- **坏的切片（横切）**：「任务 1：所有数据库层」——做完什么都验收不了，要等后面的层才知道对不对。

判据（tracer bullet）：

- 一个任务做完，能不能**独立验证 / demo** 出一点真实价值？能 → 它是合格的垂直切片。
- 不能、必须等别的任务才能验证 → 要么它是**地基任务**（见下），要么切错了方向（横切了）。

**地基任务例外**：建表 / 项目脚手架 / 公共依赖这类天然打底、无法垂直的工作，可作为**前置地基任务**单列，
被其他切片依赖。但要克制——只有真正"多个切片共用的底座"才单列，别把每一层都拆成地基任务（那就退化成横切了）。

## 二 · 任务分级：先判这个任务多大

| 档位 | 判据 | 处理 |
|---|---|---|
| **S / M（理想）** | 约 1~5 个文件、一个工作单元内能完成、单一可验收点 | 直接作为一个任务 |
| **XL（必拆）** | 跨多个关注点、说不清单一验收、5+ 文件 | 继续拆到可独立验收 |
| **过碎（必并）** | 改一行配置 / 单独加一个 import 这类 | 并入相关任务，别单列 |

**判级启发式**：

- 问"做完它，能写出一句话验收吗？"——写不出 / 要写好几条不同维度 → **太大，拆**。
- 问"它小到没有独立验收价值吗？"——是 → **太碎，并**。
- 问"它会不会和别的任务改同一批文件、必须串行？"——是 → 考虑合并，或理清依赖顺序。

## 三 · 依赖与并行：怎么排顺序

1. **建依赖关系**：标出每个任务"依赖哪些前置"（A 依赖 B 表示 B 先做）。地基任务通常是被依赖的根。
2. **依赖无环**：不能出现 A 依赖 B、B 又依赖 A。出现环 → 切片切错了，回去重切。
3. **可并行标注（轻量 wave）**：彼此无依赖、且**不改同一批文件**的任务，可并行——标在"依赖与并行视图"里。
   - 铁律：同一并行波次的任务，`涉及文件`不能冲突，否则会互相踩。
   - 这是吸收 GSD 的 wave 思想，但**轻量**：标注哪些能并行即可，不引入重型 wave 门禁。
4. **顺序可执行**：按依赖拓扑排序，任何时刻"下一个该做的任务"其前置都已完成。

## 四 · 高风险前置 + checkpoint

- **高风险 / 高不确定性前置**：把"最可能推倒重来"的任务（新技术验证、最不确定的集成）排在前面，
  早暴露问题，避免在它之上堆了一堆任务才发现方向错了。
- **checkpoint**：每 2~3 个任务设一个检查点——跑测试 / 构建 / 核心流程，确认到这里是稳的再往下。
  在"依赖与并行视图"里标出 checkpoint 位置。

## 五 · 覆盖核对：对齐 design + spec 的闸门

任务清单定稿前，把 design 的组件 / Decision 和 spec 的每条 Requirement / MUST NOT 列出来，逐条对到任务：

- design 每个**组件 / 数据结构 / Decision** → 由哪个任务实现？没有 → **落空了，补任务**。
- spec 每条 **Requirement（MUST）** → 由哪个任务的验收覆盖？
- spec 每条 **MUST NOT** → 由哪个任务确认它没发生（收口任务 / 验收项）？
- **反向扫**：任务里有没有 design / spec 没要求的工作？有 → **镀金，砍**（或退回确认是否真要）。

这是 split-task 不跑偏的核心闸门：**设计点不落空、任务不镀金**。落盘成"覆盖核对表"。

## 六 · 推断优先与假设标注

split-task 允许推断，但要让推断**可被 review 推翻**：

- 能从 design + spec + 代码库推断的**常规**拆分（哪些是地基、明显的依赖顺序、测试嵌入哪个任务）→ 推断 + **标注假设**，不打断用户。
- **会改变清单形状 / 需用户偏好**的（切片边界、任务粒度、关键依赖顺序、哪些延后到后续版本）→ 不许推断绕过，**停下问**，给推荐。
- 关键前提写成"**假设：X**"，让 review 一眼看到清单站在哪些未确认前提上。

> 检验：这个切法 / 顺序如果假设错了，要不要重排一批任务？要 → 它不该靠推断，停下确认。
