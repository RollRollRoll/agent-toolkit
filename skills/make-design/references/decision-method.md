# 决策方法 — 方案对比、分级、轻量 ADR、需求覆盖核对

> 用途：`make-design` 招牌机制的操作细则。阶段 1 分级、阶段 2 起草决策时用。
>
> **和决策探针的分工**（两层，别混）：
> - **决策探针（覆盖面）**：有**哪些**维度要决策？防"整类决策被漏"。
> - **本文（每条决策怎么做）**：选定要决策的点，怎么比较、定档、追溯、对齐 spec。

## 一 · 决策分级：先判这条决策有多重

不是每个决策都值得摆 2-3 个方案。先判级，再决定怎么处理：

| 档位 | 判据 | 处理 |
|---|---|---|
| **重大且难逆** | 改变核心结构（数据模型 / 对外契约 / 存储 / 框架），或定错代价高、难回退 | 方案对比 + Decision 块 |
| **常规可逆** | 有明显最佳实践，或定错了容易改 | 推荐 + 一句理由 |
| **需用户偏好 / 约束** | 依赖用户私有信息（偏好、预算、合规、外部依赖现状） | **停下问**，给推荐答案 |
| **可延后** | 不阻塞本次实现 | 进待解问题 |

**判级启发式**：

- 问"定错了好不好改？"——难改 → **重大**。
- 问"有没有公认最佳实践、且无特殊约束？"——有 → **常规**，直接用，别凑方案。
- 问"我有没有定这条所需的全部信息？"——缺用户私有信息 → **停下问**。

## 二 · 方案对比：重大决策怎么摆

重大决策给**至少两个真实候选**，每个写清：

- 它是什么（一两句）。
- 代价与收益（trade-off）——具体到**本项目的约束**（数据量、团队、现有栈、spec 要求），不写放之四海的教科书优缺点。
- 推荐哪个 + **为什么不是另一个**。

**铁律**：

- 候选必须是**真实可选项**，别放一个明显凑数的陪衬来衬托预设答案。
- trade-off 落到**本项目实际约束**，泛泛而谈等于没比较。
- 推断而非确认的前提，标成"**假设**"，让 review 能针对假设反驳。

## 三 · 轻量 ADR：决策怎么记下来可追溯

每条重大决策落成一个 Decision 块（轻量 ADR）：

- **标题**：决策的一句话结论。
- **候选 + 取舍 + 理由**（含"为什么不是另一个"）。
- 可选：假设、影响、待复核条件。

目的不是仪式，是让三个月后的人（包括你自己）看懂"**为什么这么定**"，不必重新趟一遍。
"重大且难逆"才值得 ADR；常规决策一行推荐就够，别把 ADR 做成负担。

## 四 · 需求覆盖核对：对齐 spec 的闸门

技术方案定稿前，把 spec 的每条 Requirement / MUST NOT 列出来，逐条对到设计落点：

- 每条 **Requirement（MUST）** → 由哪个决策 / 组件 / 数据结构满足？没有 → **落空了，补**。
- 每条 **MUST NOT** → 设计里怎么落实这个禁止？（无某入口 / 无某路径 / 不引某库）
- **反向扫**：设计里有没有 spec 没要求的功能 / 复杂度？有 → **镀金，砍**（或退回确认是否真要）。

这是 make-design 不跑偏的核心闸门：**需求不落空、不镀金**。

## 五 · 推断优先与 assumptions 标注

make-design 允许推断，但要让推断**可被 review 推翻**：

- 能从 spec + 代码库 + 最佳实践推断的**常规**决策 → 推断 + **显式标注假设**，不打断用户。
- **重大且难逆 / 需用户偏好**的 → 不许推断绕过，**停下问**。
- 所有推断出的关键前提，写成"**假设：X**"，集中可见，让 review 一眼看到方案站在哪些未确认前提上。

> 检验：这条决策如果假设错了，方案会不会塌？会 → 它不该靠推断，停下确认。
