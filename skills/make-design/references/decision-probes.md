# 技术决策探针清单 — 逼出漏掉的技术决策

> 用途：`make-design` 招牌机制的落地清单。阶段 1 逐项扫，阶段 3 回扫自检。
>
> **铁律**：每个维度都要有**结论**——决策了什么 / 为什么不适用 / 为什么延后，
> 不是一句"要考虑 X"。重大且难逆的决策，结论必须**可追溯**（候选 + 取舍 + 推荐 + 理由）。
>
> **用法**：逐项自问"这个维度本方案需要决策吗？"——
> - 需要 → 按重大 / 常规分级处理（见 [decision-method.md](decision-method.md)）。
> - 不需要 → 明确写"不适用（理由）"，别留空当忘了想。
> - 能从 spec + 代码推断的标注假设自己定；**重大难逆 / 需用户偏好**的停下问。

## 组一 · 数据与接口（系统的骨架）

### 1. 数据模型
实体 / 表 / 字段 / 关系 / 约束 / 索引；数据怎么存、怎么取。
- 要回答：核心实体是什么？字段与约束（枚举、非空、唯一）？关系与索引？
- 判级：新建核心数据模型、或改动会波及多处 → **重大**；加一个可空字段 → 常规。
- 例 Decision：用单表 + status 枚举约束，而非独立状态表——数据量小、查询简单，不值得多一张表。

### 2. 接口契约
对外接口（HTTP / CLI / 函数 API）的输入、输出、错误语义、版本与兼容。
- 要回答：暴露什么接口？输入输出形状？错误怎么表达？要不要保持向后兼容？
- 判级：对外契约（别人依赖的）→ **重大**（难改）；内部函数签名 → 常规。
- 例：只读 `GET /experiments` 返回整页 HTML，不做 JSON API——本版无外部消费者，YAGNI。

### 3. 状态管理
状态怎么存、生命周期、状态机、跨请求 / 跨组件的一致性。
- 要回答：有哪些状态？合法迁移是什么？状态存哪、谁是单一事实源？
- 判级：引入新的状态存储 / 状态机 → 重大；单字段枚举 → 常规。

## 组二 · 运行时行为（系统怎么跑、怎么扛）

### 4. 错误与失败处理
失败路径、降级、重试、用户可见的错误；对齐 spec 的边界 Scenario。
- 要回答：哪些会失败？失败了怎么办（报错 / 降级 / 重试）？用户看到什么？
- 判级：**几乎总要决策**——这是漏决策的头号重灾区。逐条对 spec 的边界 Scenario 落实现。
- 例：status 越界 → 标记数据异常而非静默显示，落实 spec「非法状态」Requirement。

### 5. 并发与一致性
资源竞争、锁、事务、操作顺序、幂等。
- 要回答：会不会有并发改同一对象？要不要事务 / 锁？重复操作幂等吗？
- 判级：写路径有并发 → 重大；纯只读 → **不适用（写明理由）**。

### 6. 性能与规模
数据量、热点、超时预算、分页 / 缓存 / 批处理。
- 要回答：数据规模多大？有没有热点路径？要不要分页 / 缓存？超时预算多少？
- 判级：明确性能要求或大规模 → 重大；小数据量内部工具 → 常可"默认不优化 + 延后"。
- 提醒：别过早优化，但要**显式判定**"本版不优化"，而不是压根没想。

## 组三 · 质量属性（容易被漏的非功能维度）

### 7. 安全与权限
认证、授权、输入校验、敏感数据、注入面。
- 要回答：谁能访问？要不要认证 / 授权？输入怎么校验？有无敏感数据 / 注入风险？
- 判级：涉及用户身份 / 敏感数据 / 外部输入 → **重大**；纯内部只读 → 可"不适用（标注假设：内部可信）"。

### 8. 可观测性
日志、指标、追踪——出了问题怎么排障。
- 要回答：关键路径要不要记日志？要不要指标？排障靠什么？
- 判级：线上服务 → 常规到重大；一次性脚本 / 小内部页 → 常"不适用"。

## 组四 · 演进与验证（落地与维护）

### 9. 迁移与兼容（brownfield 重点）
数据迁移、向后兼容、回滚、对现有行为的影响。
- 要回答：要不要迁移数据 / schema？老数据怎么办？能回滚吗？动了哪些现有行为？
- 判级：改动既有数据 / 接口 → **重大**；纯新增、不碰现有 → 常规甚至不适用。
- brownfield **几乎必答**——忽略迁移是上线事故的常见来源。

### 10. 测试策略与 seam
怎么测、关键逻辑的可测试接缝（seam）在哪、怎么对齐 spec 的成功标准。
- 要回答：spec 的成功标准怎么落成测试？关键逻辑能不能从难测外壳（IO / 框架 / 网络）里隔出来单测？
- 判级：**几乎总要决策**——测试策略是技术方案的一部分，不是事后补。
- 例：把取数 + 校验 + 排序抽成纯函数（test seam），路由只做渲染，纯函数直接单测。

### 11. 外部依赖与选型
引入哪些库 / 服务 / 框架，为什么，版本与维护风险。
- 要回答：要不要引新依赖？为什么不用现有栈 / 标准库？版本、维护活跃度、许可证？
- 判级：引入新依赖 / 框架 → **重大 + 需用户拍板**（增加维护面、可能锁定）；用现有栈 → 常规。
- 铁律：选型的 API / 版本特性**查官方文档或源码确认，不凭记忆**。

## 收尾自检

- 每个维度都有结论（决策 / 不适用 / 延后）？没有"忘了想"的空白？
- "不适用"都写了理由，没和"没想到"混为一谈？
- 重大决策都进了方案对比（候选 + trade-off + 推荐 + 理由）？
- 推断出来的决策都标注了假设？
- **错误处理 / 迁移 / 测试 seam** 这几个重灾区维度，确实逐项想过了？
