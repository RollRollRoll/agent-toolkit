# 技术设计文档模板 — 落盘用

> 用途：`make-design` 阶段 2 起草、阶段 4 落盘时用本模板。
> 默认写到仓库既有 design 位置；无既有约定时默认 `docs/design/<YYYY-MM-DD>-<主题>-design.md`。
>
> **铁律**：只到**技术决策层**——不下沉 task-by-task 步骤与验证命令（那是 writing-plans）。
> 每个重大决策都要可追溯（候选 + 取舍 + 推荐 + 理由）；spec 每条 MUST / MUST NOT 都要有设计落点。
>
> **brownfield 用 delta**：聚焦改动 + 对现有系统影响，不重述整套既有架构。

## 模板（greenfield / 全量）

```markdown
# 技术方案：<主题>

## 背景 / 目标与非目标
- 实现哪份 spec（链接 / 路径）。
- 技术上覆盖什么；显式不覆盖什么（非目标）。

## 关键技术决策（Decisions）
> 重大且难逆的决策用完整 Decision 块；常规决策一行"推荐 + 理由"即可。

### Decision: <决策标题（一句话结论）>
- 候选 A（选定）：<方案>
- 候选 B：<方案>
- 取舍：<A / B 各自的代价与收益，落到本项目的实际约束>
- 理由：<为什么选 A、为什么不是 B>
- 假设（可选）：<这条决策依赖的、推断而非确认的前提>

### 常规决策（清单）
- <决策点>：<推荐> —— <一句理由>

## 架构 / 组件 / 数据流
- 模块划分与各自职责、边界。
- 关键数据流（从入口到落地）。

## 数据模型与接口契约
- 实体 / 表 / 字段 / 约束。
- 对外接口：输入、输出、错误语义。

## 错误与失败处理
- 失败路径、降级、用户可见错误；对齐 spec 的边界 Scenario。

## 测试策略
- 怎么验证方案成立，对齐 spec 的成功标准（最好对应到"测试怎么跑"）。
- 关键逻辑的 test seam 在哪。

## 风险与权衡
- 已知风险、未验证假设、降低不确定性的手段（必要时一次性原型）。

## 需求覆盖核对
| spec 条目（Requirement / MUST NOT） | 设计落点 |
|---|---|
| <Req X> | <由哪个决策 / 组件满足> |
| <MUST NOT Y> | <设计里怎么落实这个禁止> |

## 待解问题 / Deferred
- <还没定、或显式延后的点>
```

## 模板（brownfield / delta）

```markdown
# 技术方案（delta）：<这次改什么>

## 背景 / 目标与非目标
- 实现哪份 spec（delta）；改动的技术目标；非目标。

## 现状（只写被改动触及的部分）
- 相关既有架构 / 模块 / 数据模型（不重述全系统）。
- 现状来源：<完整调研（codebase-analyzer）/ 有限扫描（标注）>

## 关键技术决策（Decisions）
### Decision: <决策标题>
- 候选 / 取舍 / 理由（同上）

## 改动设计
- 新增 / 修改 / 删除了哪些模块 / 表 / 接口。
- 与现有部分的耦合点（seam / adapter）。

## 迁移与兼容
- 数据 / schema 迁移、向后兼容、回滚考虑。
- 对现有行为的影响。

## 错误与失败处理 / 测试策略
- 聚焦改动部分，对齐 spec 的边界 Scenario 与成功标准。

## 需求覆盖核对
| spec 条目（Requirement / MUST NOT） | 设计落点 |
|---|---|
| <条目> | <落点> |

## 待解问题 / Deferred
- <还没定、或显式延后的点>
```

## 写完自查

- [ ] 重大决策都有候选 + 取舍 + 推荐 + 理由？
- [ ] 常规决策没硬凑方案？重大决策没被一笔带过？
- [ ] spec 每条 Requirement / MUST NOT 都在**需求覆盖核对**里有落点？没有镀金？
- [ ] 测试策略对齐 spec 的成功标准？
- [ ] 通篇**没下沉到 task-by-task** 步骤与验证命令？
- [ ] brownfield 用了 delta，交代了**迁移 / 兼容与对现有影响**，没重写整套架构？
- [ ] 推断出来的决策都标注了**假设**？
