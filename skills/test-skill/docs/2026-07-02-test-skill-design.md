# test-skill 技术设计

> **历史记录（2026-07-10 标注）**：本文描述最初设计，其中权限、隔离、运行器与报告路径已被 0.8.1 修订。现行契约以 `../SKILL.md` 与 `../scripts/run-headless.py` 为准。

- 日期：2026-07-02
- 状态：已确认（brainstorming 对话逐节确认通过）
- 定位：agent-toolkit 新增 skill `test-skill` 的设计依据

## 1. 背景与目标

仓库已沉淀 8 个 skill（五工序链 + codebase-analyzer + 两个支撑 skill），但 skill 写完后缺少验证手段：agent 实际执行时，是否真的按 SKILL.md 里写的指令行动？哪些指令被无视、被误解，哪些是冗余摆设？

test-skill 提供一次可复盘的测试跑，产出三样东西：

1. **逐步骤溯源**：agent 的每个动作依据 skill 中哪段指令（或没有依据、属于自由发挥）。
2. **指令覆盖统计**：skill 的每条指令被遵守 / 被违反 / 被跳过 / 未触发。
3. **设计诊断与修改建议**：失效指令的原因分析、skill 缺失指令的候选，直接可落为对 SKILL.md 的编辑。

## 2. 核心决策与理由

- **测试形态：主动发起测试跑**。给定目标 skill + 测试场景，由干净上下文的被测 agent 执行。场景可控、记录完整、不污染当前会话。（对照项"复盘当前会话的真实使用"未选：执行边界模糊、上下文互相干扰。）
- **溯源方式：盲测 + 事后溯源**。被测 agent 不知道自己在被测试，跑完后由分析者读完整执行记录做对齐。理由是观察者效应：执行者若知道每步会被检查，会比平时更严格遵守 skill，复盘结论偏乐观、失真。测出来必须是"平时表现"，才能回答"skill 写得好不好"。
  - 盲测的精确含义：**不告知测试意图**，而不是不提 skill 名。正常用户本来就会说"用 refine-idea 帮我理一理"，显式调用不破坏盲测。
- **交互回答：用户模拟器**。交互式 skill（如 refine-idea 硬性要求对话另一头有活人）在测试中提出的澄清问题，由编排者按预先生成的剧本扮演用户回答。全自动、可重复，还能刻意埋"隐藏痛点"考验 skill 的意图挖掘能力。
- **执行载体：headless claude CLI（方案 B）**。被测 agent 是 `claude -p` 进程，执行记录用 `--output-format stream-json` 落盘。理由：stream-json 是官方公开格式，稳定可依赖；测试环境与当前会话完全隔离；天然为将来批量回归铺路。（对照项方案 A"会话内子 agent + 读 `~/.claude/projects/<项目>/<会话>/subagents/agent-*.jsonl`"实现更简单、已验证记录含 thinking/tool_use/text，但依赖非公开的内部文件布局，被有意放弃；若将来 CLI 路径受阻可作为退路。）

## 3. 非目标（第一版不做）

- 不测"自然语言能否触发 skill"：测试任务中显式点名被测 skill，避免触发失败白跑。触发准确性测试留作后续演进的独立测试项。
- 不做批量回归、评分基线、多次跑方差分析。
- 不做真人实时接管回答（仅剧本模拟器）。
- 不支持非 Claude Code 平台。
- 不改动被测 skill 本身（零侵入，被测 skill 无需为可测性做任何配合）。

## 4. 形态与接口

新 skill 目录，遵循 `docs/conventions.md` 的 skill 资源约定：

```text
skills/test-skill/
  README.md          # 仓库内说明
  SKILL.md           # skill 主体，frontmatter description 按"何时触发"写法
  metadata.yaml      # id: test-skill，与目录名一致
  references/        # 溯源判定标准、复盘报告模板、模拟用户剧本模板
  docs/              # 本设计文档与后续演进记录
```

- **输入**：目标 skill（本仓库 skill 的 id / 任意 SKILL.md 路径 / 已安装 skill 名）+ 可选的测试场景描述。场景缺省时由编排者设计并请用户确认。
- **输出**：中文复盘报告一份（结构见第 9 节）+ 沙箱现场（transcript、剧本、被测 agent 产物）。
- **链路位置**：不进五工序链，是横向"质检工序"——五工序链生产 skill，test-skill 检验 skill。
- 随插件发布时按惯例更新 `.claude-plugin/plugin.json` 组件数组并递增版本。

## 5. 执行架构

两个角色：

- **编排者**：当前会话中执行 test-skill 的 agent。负责解析目标 skill、搭沙箱、逐轮驱动被测进程、扮演模拟用户、事后分析、写报告。
- **被测 agent**：headless `claude` 进程，每轮一次调用，对测试毫不知情。

```text
编排者                                被测 agent（headless claude）
  │ 1. 解析目标 SKILL.md → 编号指令清单 R1..Rn
  │ 2. 搭建沙箱（git init + 场景 fixture + 注入被测 skill）
  │ 3. 生成模拟用户剧本（用户可过目）
  │ 4. 首轮: claude -p "<测试任务>" ──────────────▶ 执行，stream-json 追加落盘
  │ 5. 读上轮输出：被测 agent 在提问/等待？
  │      └─ 按剧本生成回复 → claude -p --resume ──▶ 继续执行，追加落盘
  │    （循环，直到任务完成 / 轮数上限 / 超时）
  │ 6. 解析完整 transcript → 溯源分析
  │ 7. 产出复盘报告
```

### 5.1 指令清单化

- 读目标 SKILL.md 全文，拆成带编号的指令清单 R1..Rn。
- 粒度定义：**一条指令 = 一个可独立判定"遵守与否"的行为要求**。一个章节可拆出多条；纯背景说明、动机阐述不算指令，但记录为"背景条目"供诊断引用。
- frontmatter description 也纳入清单（其"何时不用"部分在执行中依然可判定）。
- 清单落盘沙箱 `checklist.md`，供分析阶段与报告引用。

### 5.2 沙箱

- 位置：会话 scratchpad 下建独立目录，`git init`。
- 按测试场景铺设 fixture（如测 execute-task 需预置任务清单文档）；fixture 属于场景设计的一部分，随剧本一起给用户确认。
- **被测 skill 注入**：将目标 skill 的当前版本（repo 工作区内容）复制为沙箱的项目级 skill（`沙箱/.claude/skills/<id>/`），保证测到的是开发中的最新版，而不是已安装插件里的旧版。

### 5.3 headless 多轮驱动

- 首轮（cwd = 沙箱）：

  ```bash
  claude -p "<测试任务，普通用户口吻，显式点名被测 skill>" \
    --output-format stream-json --verbose \
    --permission-mode acceptEdits \
    --allowedTools "<按被测 skill 所需的白名单>" \
    >> transcript-run.jsonl
  ```

- 从 init 事件取 `session_id`；后续每轮 `claude -p --resume <session_id> "<模拟用户回复>"`，同参数追加落盘。
- **权限**：默认 `acceptEdits` + `--allowedTools` 白名单（默认集：Read/Write/Edit/Glob/Grep + `Bash(git:*)`，按被测 skill 声明的需要增补）。不用 `--dangerously-skip-permissions`。权限拒绝导致的失败不视为测试事故，记入报告（它反映 skill 对权限的隐含要求）。
- **停止条件**（满足其一即停）：编排者判断被测 agent 已完成任务（上轮输出属于收尾陈述，而非提问或等待用户输入）；达到轮数上限（默认 10 轮）；单轮超时（默认 10 分钟，Bash timeout 兜底）。

### 5.4 用户模拟器

剧本文件在开跑前生成，包含：

- 模拟用户身份与背景；
- 真实意图，以及**隐藏痛点**——刻意不在首轮说出，检验 skill 的挖掘指令是否起作用；
- 常见问题的答案素材、回答风格（简短程度、含糊程度）；
- 边界：哪些信息不主动透露。

执行规则：编排者每轮严格照剧本回答；剧本未覆盖的问题按人设合理即兴，即兴内容追记进剧本附录，保证复盘时"用户说过什么"全部有据可查。剧本默认请用户过目，用户可选择跳过。非交互 skill 的剧本退化为"仅首轮任务"，流程不变。

## 6. 溯源分析

- **步骤重建**：解析 transcript 事件流，重建为语义步骤时间线。一步 = 一次有意图的动作（提一个澄清问题、调一次工具、给一段结论）；thinking 块用于辅助判断动作意图，本身不单独成步。
- **步骤侧判定**（每步三选一）：
  - `依据 Rx`：对应某条指令且做法相符；
  - `偏离 Rx`：对应某条指令但做法违背或变形；
  - `无依据`：skill 未要求，agent 自由发挥（可能暴露 skill 缺口，也可能是正常通用能力，由诊断环节区分）。
- **指令侧统计**（每条指令四选一）：`已遵守 / 被违反 / 被跳过 / 未触发（场景不适用）`。
  - 被跳过 = 场景适用但 agent 没做；未触发 = 场景根本没走到该指令的适用条件。
- 判定标准写成 `references/judging-criteria.md`，保证多次测试口径一致。
- transcript 过长（超出编排者可靠处理范围）时，spawn 子 agent 分段分析再汇总；阈值与分段方式写入 SKILL.md。

## 7. 错误处理与边界

- **预检**：开跑前检查 claude CLI 可用、版本支持 `stream-json` 与 `--resume`；预检失败即停并告知。
- **触发失败也是发现**：被测 agent 未调用目标 skill 时，不算白跑，作为"显式点名仍未触发"的重要发现记入报告。
- **成本确认**：一次测试是完整多轮 agent 会话，token 消耗可观；开跑前给出轮数与消耗预估，请用户确认后再启动。
- 沙箱位于会话级临时目录：复盘所需的关键信息（指令清单、剧本含即兴附录、判定明细）全部进入报告正文或附录，不依赖沙箱存活；原始 transcript 如需长期留存，报告末尾提示用户自行拷贝。

## 8. 实现前需验证的技术假设

实现计划的第一个任务就是逐条实测以下假设，每条都有失败应对，不阻塞设计：

1. `claude -p --output-format stream-json` 是否需要 `--verbose` 才输出逐轮完整消息流 → 实测确认参数组合；按实际行为调整命令模板。
2. `claude -p --resume <session_id>` 在 headless 下多轮续跑的可用性与副作用 → 实测；若不可用，退路是 `--session-id` 固定会话续跑，仍不行则退回方案 A（会话内子 agent + subagents 目录 transcript，已验证可行）。
3. 沙箱项目级 skill 与已安装同名插件 skill 的加载优先级 → 实测；若插件版遮蔽项目版，改为测试期间临时禁用对应插件，或以未占用名字的副本注入。
4. stream-json 输出中 thinking 块的可获得性 → 实测；若无 thinking，溯源分析仅基于可见动作（提问/工具调用/回复），判定标准相应收紧"意图推断"的置信度表述。

## 9. 复盘报告

**存放**：被测 skill 属于本仓库 → `skills/<被测id>/docs/test-report-YYYY-MM-DD.md`（贴合 docs/ "演进记录"定位）；第三方 skill → 当前工作目录 `skill-test-reports/`。

**结构**（全中文，模板放 `references/report-template.md`）：

1. 测试概要 —— 目标 skill 与版本、场景、剧本要点、轮数、时长、沙箱路径。
2. 步骤溯源表 —— 时间线上每步：步骤描述、判定（依据 Rx / 偏离 Rx / 无依据）、说明。
3. 指令覆盖统计 —— 指令总数、遵守率、四类状态的分组清单。
4. 设计诊断 —— 被违反/被跳过指令的失效原因（表述模糊？埋得太深？与其他指令冲突？）；未触发指令是场景未覆盖还是本身冗余；"无依据"步骤聚类出 skill 缺失指令的候选。
5. 修改建议 —— 按优先级排列，措辞落到"对 SKILL.md 的具体编辑动作"。
6. 附录 —— 沙箱产物清单、剧本（含即兴附录）、原始 transcript 路径（会话级临时目录，长期留存需自行拷贝）。

## 10. 验收方式

1. **冒烟**：用 test-skill 测本仓库一个交互式 skill（首选 refine-idea）走通全流程，产出完整报告。
2. **报告质量三查**：溯源表每步都有判定；指令清单每条都有状态；修改建议可直接执行。
3. **反事实检查**：在某 skill 副本里故意塞入一条显眼指令（如"开始前必须先输出某标记"），跑一次测试，验证该指令被违反时报告确实捕捉到——防止分析走过场。
