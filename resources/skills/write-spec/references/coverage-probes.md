# 双探针清单 — 逼出漏掉的需求

> 用途：`write-spec` 招牌机制的落地清单。阶段 2 起草时逐项扫，阶段 3 回扫自检。
>
> **铁律**：探针的产物必须是**可验证的** Scenario 或 MUST NOT，不是一句提醒。
> 每命中一项，就把它写成 `WHEN→THEN` 或 `MUST NOT + 怎么验证`。
>
> **用法**：不是每项都要写——逐项自问"这条适用吗？"，适用的才写；
> 不适用的，在自检时能说出"为什么不适用"即可。

## 探针一：边界覆盖（Edge Coverage）

逐项自问"适用吗？"，适用的写成 Scenario：

### 1. 边界值
最大 / 最小 / 刚好越界的输入。
- 转 Scenario 例：WHEN 输入长度 = 上限 + 1 → THEN MUST 拒绝并提示。

### 2. 空值
空输入 / 空集合 / null / 缺字段。
- 例：WHEN 清单为空 → THEN MUST 显示空态，MUST NOT 报错。

### 3. 排序
顺序是否影响结果；相等元素的稳定次序。
- 例：WHEN 两条记录排序键相同 → THEN MUST 有稳定次级排序，刷新不乱序。

### 4. 并发
同时操作同一对象 / 资源竞争。
- 例：WHEN 两个请求同时改同一条 → THEN 结果 MUST 是其一，MUST NOT 出现各改一半。

### 5. 幂等
重复执行同一操作。
- 例：WHEN 同一创建请求重试两次 → THEN MUST 只产生一条，不重复。

### 6. 精度
浮点 / 金额 / 时间 / 单位精度。
- 例：WHEN 金额计算涉及分 → THEN MUST 不出现浮点误差导致的 ±0.01。

### 7. 编码
字符集 / 转义 / 大小写 / 二进制 / 多语言。
- 例：WHEN 输入含 emoji 或中文 → THEN MUST 正确存储与展示，不乱码。

### 8. 超时与失败
依赖不可用 / 超时 / 部分失败。
- 例：WHEN 依赖服务超时 → THEN MUST 在 N 秒内返回明确错误，MUST NOT 无限挂起。

> 全不适用是可能的（如纯只读静态展示），但**必须是自问后的结论**，不是忘了扫。

## 探针二：禁止覆盖（Prohibition Coverage）

把概念 / 讨论里每条"不做什么"逐条转成**可验证的负向需求**：

1. 取出概念单「不做什么」里的每一条。
2. 改写成 `系统 MUST NOT <行为>`。
3. **补上"怎么验证它没发生"**——这一步是关键，写不出验证方法的，说明边界还太虚。

例：

- `不做自动同步` → `系统 MUST NOT 从 CI / 外部平台拉取或修改状态` →
  验证：代码无外部同步入口、无定时任务、UI 无同步配置项。
- `不做界面内编辑` → `系统 MUST NOT 提供新增 / 编辑入口` →
  验证：无写库路由、UI 无表单与编辑按钮。
- `首版不做多租户` → `系统 MUST NOT 引入租户隔离行为` →
  验证：无租户切换入口；接口不接受租户选择参数；同一用户视角下不存在按租户隔离的数据可见性差异。

> 为什么单列这个探针：一半的实现偏差来自对"不该做什么"的沉默假设。
> 把它变成负向验收，实现期就能逐条核对"我没多做"。

## 收尾

两个探针扫完，回到自检清单确认：

- 边界探针逐项有结论（写了 / 明确不适用）。
- "不做什么"每条都成了带验证方法的 MUST NOT。
