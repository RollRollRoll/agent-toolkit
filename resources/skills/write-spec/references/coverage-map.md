# 能力覆盖图（Coverage Map）— greenfield 防漏与防膨胀

> 用途：`write-spec` 在 **greenfield（新增能力）** 阶段，系统性回答"这个功能 / 领域
> 有哪些能力维度、本版各做到什么程度"，让全面性**可被 review、漏项可被 catch**。
> brownfield（改既有功能）走 delta，**不用**本清单——改动范围已被既有系统框定。
>
> **和双探针的分工**（两层，别混）：
> - **Coverage Map（宏观全面性）**：有**哪些**能力维度？每个维度本版做不做？防"整类功能被漏"。
> - **双探针（微观完整性）**：选定**要做**的需求，它的边界场景与"不做什么"。防"选定需求的边界被漏"。
>
> 顺序是一个漏斗：**先用 Coverage Map 圈定做哪些维度，再对纳入的维度用双探针抠边界。**

## 步骤一 · 功能空间调研（先问后调，可跳过）

greenfield 最大的漏洞是"没想到的整类功能"。先判断本次是否属于
**有成熟功能空间、容易漏整类能力**的领域（如权限 / 通知 / 购物车 / 导入导出 / 评论）：

- **是，且值得调研** → **先问用户**：「要不要先做一轮功能空间调研，把这个领域的常见
  能力维度摸一遍，避免漏整类功能？」**用户同意才调。**
- 用户同意 → 用 **deep-research** skill 调研，问题聚焦三件事：
  1. 这个领域常见有哪些**能力维度**？
  2. 按产品丰富度分哪些**档**（如 MVP / 标准 / 完整）？
  3. 哪些维度**容易被漏、但会改变需求边界 / spec 形状**？
  - 问题模板：`<X 类产品 / 功能> 通常包含哪些能力维度？按丰富度（MVP / 标准 / 完整）`
    `如何分档？哪些维度容易被遗漏、却会显著改变需求边界或数据模型？`

**REQUIRED SUB-SKILL**（用户同意调研时）：deep-research。

> **铁律**：调研产出是**功能空间参考，不是需求**——一项都**不默认纳入**，全部交给步骤三
> 的 Coverage Map 逐条定档。把调研当需求、所有维度全做，正是膨胀的来源。

**跳过的情形**（说明理由即可，仍要凭需求 + 自身知识自列维度）：

- 小工具 / 一次性脚本，功能空间有限。
- 高度新颖、无成熟产品可参照。
- 需求已极详尽，维度已被列全。

## 步骤二 · 列能力维度

把两个来源合并成一张维度清单：

- **需求已含**的维度（用户明说要做 / 不做的）。
- **调研 / 自列补充**的维度（做了调研就用调研结果；跳过调研则凭自身领域知识自列）。

常见维度**提示**（按需取用，是提示不是清单义务，别硬凑）：
核心数据与操作（增删改查）、查询 / 过滤 / 排序、状态与生命周期、权限与可见性、
空态 / 错误 / 失败处理、批量与规模、导入 / 导出 / 集成、审计与历史、通知与提醒、
配置与偏好、国际化、可访问性、并发与一致性。

## 步骤三 · 给每个维度落一个档位（防膨胀核心）

每行**必须**落唯一一档，**不允许"全都纳入"**：

- **纳入**：本版要做 → 阶段 2 写成 Requirement + Scenario。
- **默认不做**：本版保守排除 → 写成 MUST NOT（或在覆盖表标注），**可被 review 反对**。
- **后续版本**：本版不做但记下 → 进覆盖表"后续"+ spec 待解问题。
- **必须确认**：无法默认、且会改变 spec 结构 → 走步骤四提问，问完归入前三档之一。

**判档启发式**：

- 需求明确要 → **纳入**。
- 需求明确不要 → **默认不做**（写成 MUST NOT）。
- 没提、缺了它本版照样成立、加上也不改结构 → **默认不做** 或 **后续版本**。
- 没提、但做不做会**改变 spec 结构 / 核心数据模型 / 主流程** → **必须确认**。

## 步骤四 · 把"必须确认"分三级（别一次问几十个）

只有会**卡住起草**的才打断用户。每个待澄清点归一类：

- **现在问**：答案会改变 spec 结构（核心数据模型 / 主流程 / 范围硬边界）
  → 阶段 1 聚焦提问，一次一问或小批量。
- **采用默认**：不影响主方向 → 记一句"采用保守默认：X"，继续，不打断。
- **放 Open Questions**：重要但不阻塞本版 → 进 spec「待解问题」。

> 检验：「这个问题不问，spec 能不能起草？」能 → 它**不是**"现在问"。

## Coverage Map 模板（落盘进 spec，精简版）

> 落盘表**只含已收敛的三档**：纳入 / 默认不做 / 后续版本。"必须确认"是起草期的过程状态，
> 落盘前必须已问完并归入这三档之一，确认过程写进"说明"列（如"经确认后纳入：<答案>"）。

| 能力维度 | 来源 | 档位 | 说明 / 去向 |
|---|---|---|---|
| <维度> | 需求 | 纳入 | → Requirement: <标题> |
| <维度> | 调研 | 默认不做 | 见 MUST NOT |
| <维度> | 调研 | 后续版本 | <为什么放后续> |
| <维度> | 需求 | 纳入 | 经确认后纳入：<答案> |

> 落盘**精简版**即可：重点让 review 一眼看到"哪些做、哪些不做、为什么"。

## 收尾自检

- 每个维度都有**唯一档位**？没有"待定"残留？
- "纳入"的都进了 Requirement？
- "默认不做"的关键项都进了**可验证的 MUST NOT**？
- "必须确认"的都已问、或已归入待解问题？
- **没有**把调研里所有维度无脑纳入（膨胀）？
