# Write Spec — 行为规格 Skill

## 你的任务

把一个**边界已经清晰的想法 / 需求**（理想情况下来自 refine-idea 的概念单），
转化为一份**可被验证的行为规格**。你要把模糊的"应该能用"钉成精确的四件事：

1. **系统该有什么可观察行为** —— 用 Requirement + MUST 表达，每条都配 Scenario（WHEN→THEN）。
2. **明确必须不发生什么** —— 把"不做什么"转成 MUST NOT 负向需求 + 可验证检查。
3. **边界场景下如何表现** —— 边界值 / 空值 / 并发 / 幂等…… 逐项探，适用的写成 Scenario。
4. **怎么算做完了** —— 可执行的成功标准，不是"能用就行"。

**刻意止步于此**：不做技术选型、不画架构、不拆任务、不排期——那是下游 write-plan 的职责。
你的产出是一份"行为说清楚了、验收定死了、可以放心进入实现"的规格，而不是实现方案。

> 一句话：把"我们要解决 Y、做 A/B、不做 C/D、成功标志 Z"的概念，钉成"系统 **MUST** 这样、
> **MUST NOT** 那样、在这些边界下表现如此、用这些可执行检查验收"的规格。

## 核心理念

spec 是**编码前的最后一道闸门**。需求里每一处没说清的行为、每一个没被表态的边界、
每一条沉默的"不做什么"，都会在实现期变成返工、扯皮或上线事故。最便宜的消除时机，
是在 plan 和 code 存在之前——把它写成可被验证的契约，让"做完没做完"不再靠口头判断。

write-spec 不挖意图（那是 refine-idea 的事），也不给方案（那是 write-plan 的事）。
它只做一件事：**把已清晰的概念，翻译成不会产生第二种理解的行为契约。**

## 何时用 / 何时不用

**适用**：

- 概念已清楚（或手上有 refine-idea 概念单），准备进入实现前，需要一份可验证规格。
- 要新增一个功能 / 能力，或要改既有功能，需要先写清"改成什么样、怎么验收"。
- 协作中需要一份"实现期可对照、验收时可逐条核对"的依据。

**不适用**（别硬套）：

- 概念还模糊、意图没挖透 → 先用 refine-idea。
- 要的是技术选型 / 架构 / 任务拆解 → 那是下游 write-plan。
- 已在写代码、定位 bug、做评审。

**⚠️ 加载约束（硬性）**：本 skill 的产出必须经一个**能实时 review 的活人**确认才放行。
禁止在 CI / 定时任务 / 自治循环等非交互场景里把它当作"自动产 spec"的工具——那种场景遇到
"没有 spec"，正确做法是**把它标记为 blocker 并停下**，而不是自动生成一份没人 review 的规格往下跑。

## 招牌机制：行为契约 + 双探针 ⭐

这是 write-spec 区别于"随手写个需求文档"的核心。一份 spec 不是把需求换种说法抄一遍，
而是让每条需求都过两关：

### 关一：可验证性

每条 Requirement 必须能写出至少一个 **Scenario**：`WHEN <触发条件> → THEN <可观察的结果>`。

- 写得出可观察结果 → 它是需求。
- 写不出（只能说"系统应该很好用 / 性能要好"）→ 它是愿望，不是需求。**打回重写成可观察的，或删掉。**

> 检验：把 Scenario 交给一个不懂业务的人，他能照着判断"通过 / 不通过"吗？不能，就还没写清。

### 关二：双探针强制覆盖

光写"要做什么"会漏掉一半的真实需求——漏在边界上，漏在"不该发生什么"上。
两个探针逐项扫，把漏的逼出来：

**边界探针（Edge Coverage）** —— 逐项自问"这条适用吗？"，适用的必须配 Scenario：

- 边界值（最大 / 最小 / 刚好越界）、空值（空输入 / 空集合 / null）、排序（顺序是否影响结果）、
  并发（同时操作同一对象）、幂等（重复执行结果是否一致）、精度（浮点 / 金额 / 时间）、
  编码（字符集 / 转义 / 二进制）、超时与失败（依赖不可用时）。

**禁止探针（Prohibition Coverage）** —— 把概念里每条"不做什么"转成可验证的负向需求：

- `不做自动同步` → `系统 MUST NOT 从 CI / 外部平台拉取或修改状态` →
  验证：`代码中不新增外部同步入口；UI 无同步配置项`。
- 关键：负向需求也要可验证。写不出"怎么检查它没发生"的，就还没转化完。

（两个探针的完整清单与"怎么把每一项转成 Scenario / MUST NOT"，见 [references/coverage-probes.md](references/coverage-probes.md)。）

## 工作流程（阶段 0 + 四个工作阶段）

### 阶段 0 · 定位与上下文

1. **扫项目**：用 Glob / Grep / Read 摸清结构、**测试怎么跑**（决定"成功标准"能落成什么形式的检查）、
   已有 spec / 文档、相关代码。脱离代码库写出来的 spec 是愿望清单。
2. **接上游**：若存在 refine-idea 概念单（`docs/ideas/*` 或用户指明的位置），读入作为输入；
   没有，就做一次**轻量**意图确认（确认"要解决什么、不做什么"够清楚即可，不重复 refine-idea 的深挖）。
3. **判 greenfield / brownfield**：
   - 新增能力 → 写全量 spec。
   - 改既有功能 → 用 **delta**：只写 `ADDED Requirements` / `MODIFIED Requirements` / `REMOVED Requirements`，不重写整个系统。
4. **判规模**：若这其实是多个独立能力，**先提示拆分**，逐个写 spec，别把一份 spec 撑成全系统说明书。

### 阶段 1 · 确认关键边界（先确认，再起草）

起草前，只就**答案能改变 spec 形状**的少数关键点做确认式提问：

- 关键验收口径（"做完"到底以什么为准）。
- 边界探针里暴露出的、需要用户拍板的取舍（如"并发时以谁为准""空输入算错误还是默认值"）。
- 范围的硬边界（哪些显式不做）。

一次一问或小批量；**不重复 refine-idea 的意图挖掘**——意图已经清楚，这里只钉行为与验收。

### 阶段 2 · 起草 spec（行为契约）

按 [references/spec-template.md](references/spec-template.md) 写：

- **背景 / 范围**（简短）：为什么做、覆盖什么、显式不覆盖什么。
- **行为需求**：Requirement + MUST/SHALL + 每条配 Scenario（WHEN→THEN）。brownfield 用 delta 标注。
- **不做什么（MUST NOT）**：禁止探针的产物，每条配可验证检查。
- **边界场景覆盖**：边界探针的产物。
- **成功标准**：可执行的检查（最好能对应到"怎么跑测试"）。
- **待解问题**：还没定、但需要记录的点。

**纯行为**：不写技术选型 / 架构 / 数据模型 / 任务（**行为级约束例外**——如
"MUST NOT 引入外部网络调用"这种可验证的行为约束，属于 spec；"用 Redis 做缓存"这种技术选型，不属于）。

### 阶段 3 · 自检（双重，落盘前过一遍）

**① spec 卫生**（借 superpowers）：

- 占位符 / TODO / 没写完的节？补全。
- 自相矛盾？（两条需求打架、Scenario 与 Requirement 不一致）
- 范围过大？需要拆成多份？
- 歧义？（任何一条能有两种解读 → 选一种写死）

**② 覆盖自检**（双探针回扫）：

- 每条 Requirement 都有 Scenario 吗？
- 边界探针逐项过了吗？适用的都写了吗？
- 概念里每条"不做什么"都转成 MUST NOT 了吗？
- 成功标准是可执行的吗？

就地修复，不必重复 review。

### 阶段 4 · 用户 review 门禁（硬门）

1. **落盘**：把 spec 写入仓库（默认就写文件——spec 是要进仓库、实现期持续更新的工件）。
   位置见下文「产物与落盘」。
2. **请用户 review**，并明确这条规矩：
   > **未通过 review 的 spec，不进入 plan / code。**
3. 用户要改 → 回阶段 2 / 3 修订并重过自检；用户**显式确认** → 过门。
4. **过门后**：可以一句话提示"需要的话，可据此进入 write-plan（技术方案 + 任务拆解）"，
   但**不自动调用下游、不开始写代码**。本 skill 到"规格确认"为止。

**哪些回答不算 "yes"**（同 refine-idea 的门禁纪律）：

- "你看着办 / 都行" = 委托，不是决策。
- "听起来不错" = 模糊回应。
- 沉默后一句"那就开始吧" = 放弃，不是确认。

## 产物与落盘

- **默认单份文档**，内容是行为契约（Requirement / MUST / Scenario / MUST NOT / 边界 / 成功标准）。
- **位置**：跟随仓库既有 spec 约定——若已有 `docs/specs/` 或 OpenSpec 式布局，就跟随；
  否则默认 `docs/specs/<YYYY-MM-DD>-<主题>-spec.md`（目录不存在先跟用户确认）。
- **brownfield** 用 delta，只写这次改了什么。
- 规模极大 / 多能力时，**可提议**拆成 per-capability 多文件，但这是 scale-up，不是默认——别给小改动套重型结构。

## 核心原则（不要违反）

1. **行为契约，不是需求复述** —— 每条需求都要能写出可观察的 Scenario，否则是愿望。
2. **先扫上下文，再写 spec** —— 脱离代码库（尤其"测试怎么跑"）的 spec 是愿望清单。
3. **双探针是强制项** —— 边界与"不做什么"是漏需求的两个重灾区，逐项扫，别省。
4. **"不做什么"要可验证** —— MUST NOT 也得能检查它没发生，否则没转化完。
5. **纯行为，不越界到方案** —— 技术选型 / 架构 / 任务留给 write-plan；只有"行为级约束"留在 spec。
6. **未过 review 不放行** —— spec 是编码前闸门，显式确认才进 plan / code。
7. **止步于规格，不自动往下** —— 不调用下游 skill、不写代码。

## 终止条件（可检验）

- **达标即停**：当每条 Requirement 都有 Scenario、双探针都过、"不做什么"都成了可验证 MUST NOT、
  成功标准可执行——spec 就够了，进入 review 门禁，别为"更完整"无限加需求。
- **兜底**：如果发现概念本身还没清（写到一半发现"要解决什么"都说不清），**停下来退回 refine-idea**，
  别用 spec 的外壳硬包一个没想透的想法。

## 自检清单（落盘前过一遍）

- [ ] 每条 Requirement 都配了**可观察的 Scenario**（WHEN→THEN）？
- [ ] **边界探针**逐项扫过，适用的都写成 Scenario 了？
- [ ] 概念里每条**"不做什么"**都转成了可验证的 **MUST NOT**？
- [ ] **成功标准**是可执行的检查，不是"能用就行"？
- [ ] 通篇**没有技术选型 / 架构 / 任务拆解**（行为级约束除外）？
- [ ] brownfield 用了 **delta**（ADDED/MODIFIED/REMOVED），没重写全系统？
- [ ] 没有**占位符 / 自相矛盾 / 歧义**？范围**不过大**？
- [ ] 动笔前**扫过项目上下文**（结构 / 测试方式 / 已有文档）？
- [ ] 拿到的是**显式 review 确认**，不是"你看着办 / 听起来不错"？
- [ ] 全程**止步于规格**，没自动进入 plan / 没写代码？

## 一次完整示范（照着这个节奏走）

> 接 refine-idea 那个"实验清单"概念，看它怎么被钉成可验证规格。

**输入（来自 refine-idea 概念单）**

> 要解决：团队没有权威的实验清单。做：只读展示实验 + 状态 + 负责人。
> 不做：趋势图表、界面内编辑、CI 自动同步。成功：打开就能看到全部实验。

**阶段 0 · 定位与上下文**

> 扫了项目：Flask + SQLite 小应用，测试用 pytest。概念单齐全，直接接住。
> 这是新增能力 → 全量 spec。

**阶段 1 · 确认关键边界（聚焦提问）**

> 起草前两个会改变 spec 形状的点想跟你确认：
> 1. "实验状态"是固定枚举（draft/running/done）还是自由文本？
> 2. 清单为空时，是显示空态提示，还是算异常？

**用户**：状态固定三种枚举；空了就显示"暂无实验"。

**阶段 2 · 起草（节选）**

> ### Requirement: 实验清单只读展示
> 系统 MUST 在一个页面列出所有实验的名称、状态、负责人，按创建时间倒序。
> #### Scenario: 正常展示
> - WHEN 库中有 3 条实验
> - THEN 页面 MUST 列出全部 3 条，每条含名称 / 状态 / 负责人，最新的在最上
> #### Scenario: 空清单
> - WHEN 库中无实验
> - THEN 页面 MUST 显示"暂无实验"，且 MUST NOT 报错
>
> ### Requirement: 状态取值约束
> 实验状态 MUST 限定为 draft / running / done 三种枚举之一。
> #### Scenario: 非法状态
> - WHEN 数据中出现枚举外的状态值
> - THEN 系统 MUST 标记为数据异常（不静默显示原始值）
>
> ## 不做什么（MUST NOT）
> - 系统 MUST NOT 提供任何在界面内新增 / 编辑实验的入口 —— 验证：UI 无表单 / 编辑按钮；无写库路由。
> - 系统 MUST NOT 从 CI / 外部实验平台拉取或同步状态 —— 验证：代码无外部同步入口；无定时任务。
> - 系统 MUST NOT 渲染任何趋势 / 对比图表 —— 验证：页面无图表组件 / 容器，无引入图表库。
>
> ## 边界场景覆盖
> - 空值：空清单（见上）；负责人字段为空时 MUST 显示"未指派"而非空白。
> - 排序：创建时间相同的两条，MUST 有稳定的次级排序（按 id），不得每次刷新乱序。
>
> ## 成功标准
> - `pytest` 下新增测试覆盖：3 条正常展示、空清单、非法状态、负责人为空——全绿。
> - 人工：打开页面无需追问任何人即可看到全部实验 + 状态 + 负责人。

**阶段 3 · 自检**

> 每条 Requirement 有 Scenario ✓；边界探针扫了空值 / 排序，并发 / 幂等 / 精度不适用（只读）；
> 三条"不做什么"都成了可验证 MUST NOT ✓；成功标准对应 pytest ✓。

**阶段 4 · review 门禁**

> spec 已写入 `docs/specs/2026-06-17-experiment-list-spec.md`。请 review；未过 review 不进入 plan / code。
>
> 〔用户显式确认后过门，提示"可据此进 write-plan"，不自动往下。〕

## 反例（不要这样做）

❌ 把概念单换种措辞抄一遍当 spec —— 没有 Scenario、不可验证。
❌ 写"系统应该性能好 / 体验流畅"这种无法判断通过与否的"需求"。
❌ 只写"要做什么"，跳过边界探针和"不做什么"——漏掉一半真实需求。
❌ "不做什么"只写一句"不做同步"，没给"怎么验证它没发生"。
❌ 在 spec 里写"用 React + SQLite""加一个 Redis 缓存层"——越界到技术方案。
❌ 改一个小功能却重写整个系统的 spec —— 该用 delta。
❌ 动笔前不看代码、不看测试怎么跑，写出一份脱离代码库的愿望清单。
❌ 没落盘、没让用户 review 就说"spec 好了，开始写代码"。
❌ 概念其实还没想清，却用 spec 的外壳硬包着往下冲 —— 该退回 refine-idea。
❌ 在 CI / 自治循环里自动产 spec，没有活人 review。
