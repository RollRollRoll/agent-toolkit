# Finish Branch — 开发收尾 Skill

## 你的任务

一段开发**完成且测试通过**后，给当前分支 / worktree **收尾**：确认测试、看清状态、给出
"合并 / 本地保留 / 创建 PR / 丢弃"的选项，把**破坏性与对外操作交用户拍板**，最后清理。

这是 agent-toolkit 的**执行阶段支撑 skill**，通常由 execute-task 执行完后调用，也可被任何"开发完了、
该怎么收口"的场景单独复用。它负责"收尾决策 + 清理"——**建立工作区交 setup-worktree**（职责分离）。

> 一句话：代码写完、测试绿了，在这里决定这段工作的去向（合并 / 留着 / PR / 丢弃），并收拾干净。

## 何时用 / 何时不用

**适用**：

- 一段开发完成、测试通过，要决定分支怎么处理。
- execute-task 整体验收通过后的收尾。
- 在 worktree 里干完活，要合并回去并清理 worktree。

**不适用**：

- 开发还没完 / 测试还没过 —— 先回去做完、修绿，别提前收尾。
- 要建立隔离工作区 —— 那是 setup-worktree。

## 核心流程（验证 → 检测状态 → 给选项 → 清理）

1. **验证测试全绿**：跑测试确认通过。**不绿不收尾**——回去修，别带着红测试收口。
2. **检测 repo / worktree 状态**：
   - 当前在哪个分支？是否在 worktree？
   - 有无**未提交改动**？有 → 先处理（提交，或明确告知用户）。
   - 与主干的关系（领先 / 落后多少 commit）。
3. **给出收尾选项（交用户拍板）**：
   - **合并到主干**：merge 回主分支。
   - **本地保留**：留在分支，暂不合并。
   - **创建 PR**：推送并开 PR。
   - **丢弃**：放弃这段工作（删分支 / worktree）——**不可逆**。
4. **执行用户选择 + 清理**：
   - worktree 用完 → 清理（如 `git worktree remove`）。
   - 临时分支、遗留的调试代码 / 打印 → 清理。

## 注意事项与危险操作 gate

- **⚠️ 破坏性 / 对外操作必须用户明确确认，不自动做**：合并、push、创建 PR、丢弃——呼应 CLAUDE.md 危险操作机制。
- **测试没过不进收尾**：先回去修绿。
- **丢弃不可逆**：删分支 / worktree 前**二次确认**。
- **先检测再动手**：状态没看清（在错分支、漏了未提交改动）就操作，容易误伤。

> 检验：你要执行的收尾动作里，有没有"删 / 合并 / push / 开 PR"这种一旦做了不好撤的？有 → 先让用户拍板。

## 反例（不要这样做）

❌ 测试没过就收尾 / 合并。
❌ 自动合并 / 自动 push / 自动开 PR —— 没让用户拍板（危险操作）。
❌ 丢弃（删分支 / worktree）不二次确认就执行。
❌ 不检测状态就操作 —— 在错的分支 / 漏掉未提交改动。
❌ 在这里建立工作区 —— 越界到 setup-worktree。

## 相关

- 通常由 **execute-task**（阶段 4 收尾）在整体验收通过后调用。
- 建立隔离工作区见 **setup-worktree**。
