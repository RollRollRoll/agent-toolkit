# 拆分探针清单 — 逼出漏掉的任务

> 用途：`split-task` 招牌机制的落地清单。阶段 1 逐项扫，阶段 3 回扫自检。
>
> **为什么**：照着技术设计里写到的模块拆任务，只覆盖"设计明说的部分"；但功能能不能上线，
> 往往取决于设计不显眼却必须做的工作——迁移、测试、配置、回滚、可观测。漏一个，就会"代码写完却上不了线 / 出事故回不了滚"。
>
> **铁律**：每个维度都要有**结论**——拆了什么任务 / 为什么不适用 / 为什么延后，不是一句"要注意 X"。
>
> **用法**：逐项自问"这个维度本次实现需要单独的任务吗？"——
> - 需要 → 列成任务（按垂直切片 / 地基归位，见 [slicing-method.md](slicing-method.md)）。
> - 不需要 → 明确写"不适用（理由）"，别留空当忘了想。
> - 能从 design + spec 推断的标注假设自己定；**会改变清单形状 / 需用户偏好**的停下问。

## 组一 · 数据与契约（让功能能存数据、能被调用）

### 1. 数据迁移
建表 / 改 schema / 数据回填 / 索引。
- 要回答：要新建或改动表 / 字段吗？老数据要回填或转换吗？要加索引吗？
- 判级：有 schema 变更或数据回填 → **需任务**（通常是地基前置）；纯逻辑、无数据变更 → 不适用。
- 例：T1 建 `experiments` 表 + status 的 CHECK 约束。

### 2. 种子数据 / 初始数据
演示数据、默认配置行、初始化脚本。
- 要回答：功能要不要初始 / 演示数据才能跑起来或被验收？
- 判级：需要初始数据才能验证 → 需任务（小，常并入建表任务）；不需要 → 不适用。
- 例：建表任务里插几条实验演示数据，供展示页验收。

### 3. 接口契约落地
HTTP 路由 / CLI 命令 / 对外函数 API 的实现任务。
- 要回答：design 定的对外接口，落成哪个 / 哪些实现任务？
- 判级：有对外接口 → **需任务**；纯内部重构、无新接口 → 视情况。
- 例：T3 `GET /experiments` 路由。

## 组二 · 质量保障（让功能可信、可回退）

### 4. 测试
单元 / 集成 / 端到端测试，对齐 spec 成功标准。
- 要回答：spec 的成功标准怎么落成测试？测试是**嵌入**各任务的验证方式，还是需要**单独**的测试任务？
- 判级：**几乎总要**——但优先作为每个任务的"验证方式"嵌入（关一），只有大块独立测试工程才单列任务。
- 例：`list_experiments()` 的 pytest 用例作为 T2 的验证方式，不另立纯测试任务。

### 5. 回滚 / 降级
出问题怎么退回、灰度 / 开关、破坏性操作的可逆性。
- 要回答：这次改动可逆吗？要不要迁移回滚脚本 / feature flag / 灰度？
- 判级：有破坏性迁移 / 高风险变更 → **需任务**；纯新增、只读、无破坏 → 不适用（写明理由）。
- 例：实验清单是纯新增只读功能、无破坏性迁移 → 回滚不适用。

### 6. 可观测性
日志 / 指标 / 告警 / 追踪。
- 要回答：关键路径要不要加日志 / 指标？出了问题靠什么排障？
- 判级：线上服务 / 关键写路径 → 常需任务；一次性脚本 / 只读静态页 → 常不适用。
- 例：只读静态展示，无需埋点 → 不适用。

## 组三 · 环境与交付（让功能装得上、跑得起、发得出）

### 7. 配置与环境变量
新增配置项 / 密钥 / 环境变量 / 配置说明。
- 要回答：要新增配置或环境变量吗？各环境（本地 / CI / 线上）都要配吗？
- 判级：有新配置 → 需任务（含告知怎么配）；无新配置 → 不适用。

### 8. 依赖安装
新增三方库 / 服务 / 系统依赖的引入任务。
- 要回答：design 引入了新依赖吗？要不要更新 lock 文件 / 安装说明？
- 判级：引入新依赖 → 需任务；沿用现有栈 → 不适用。
- 提醒：依赖**该不该引入**是 make-design 的决策；这里只落地"安装 / 锁版本"的任务，不重做选型。

### 9. 构建 / CI
构建脚本 / CI 流水线 / 部署配置的改动。
- 要回答：要改构建步骤吗？CI 要加新的测试 / 检查吗？部署方式变了吗？
- 判级：有构建 / CI / 部署变更 → 需任务；沿用现有流程 → 不适用。
- 例：沿用现有 pytest、无新构建步骤 → 不适用。

## 组四 · 集成与协作（让功能接得上、说得清）

### 10. 集成点
与现有模块 / 外部系统的对接、seam / adapter。
- 要回答：这次改动要和哪些现有部分或外部系统对接？接缝在哪？
- 判级：跨模块 / 跨系统对接 → 需任务；自包含 → 不适用。

### 11. 文档
README / API 文档 / 使用说明 / 变更记录。
- 要回答：要不要更新文档让别人会用 / 知道改了什么？
- 判级：对外能力 / 团队协作 → 常需任务；内部一次性小工具 → 常延后或不适用（写明）。
- 例：内部小工具，本版文档延后。

## 收尾自检

- 每个维度都有结论（需任务 / 不适用 / 延后）？没有"忘了想"的空白？
- "不适用"都写了理由，没和"没想到"混为一谈？
- **数据迁移 / 测试 / 回滚 / 配置** 这几个重灾区维度，确实逐项想过了？
- 推断出来的"需任务 / 不适用"判定，假设都标注了？
