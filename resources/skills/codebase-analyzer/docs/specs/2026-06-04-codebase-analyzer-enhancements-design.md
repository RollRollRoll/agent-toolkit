---

## 1. 背景与目标

`codebase-analyzer` 是一个产出**中文项目调研报告**的 skill，当前由 7 个文件组成：

- `SKILL.md` —— 主流程（输出深度三档 + 6 步工作流 + 大项目策略）
- `references/template-quick.md` / `template-standard.md` / `template-deep.md` —— 三档报告模板
- `references/investigation-checklist.md` —— 三档通用追问清单
- `references/project-type-checklists.md` —— 6 类项目专项（CLI / Web / SDK / Agent / MCP / DevTool·Workflow）
- `references/parallel-strategy.md` —— 大项目"按模块"并行策略

本次围绕三个改进诉求做增强：

1. **增加「项目亮点」输出** —— 报告要点名"这个项目最值得学习/借鉴的设计"。
2. **Workflow 工具穷举工作场景**（仅深度档）—— 对开发者流程工具，把"挑 1~2 条代表性流程"升级为"穷举所有工作场景 + 每场景配完整流程"。
3. **Claude Code 扩展件完整解读 + 并行**（仅深度档）—— 对含 skill / slash command / hooks / subagent 的项目，逐条画像 + 画整体装配图，并支持子 agent 并行以防上下文过长。

## 3. 诉求 1：项目亮点

### 3.1 概念界定（关键，防止与"优点"混淆）

- **亮点** = **技术欣赏 / 学习借鉴视角**："这个设计很妙，值得我学" —— 精巧的抽象、独特的算法、干净的扩展机制、聪明的工程取舍。
- **优点** = **决策视角**："质量不错，可以放心用 / 二开" —— 测试完善、文档齐全、社区活跃。

两者**并入同一个综合评价章节**，亮点作为**最醒目的子项置顶**。亮点每条必须落到具体 `path:line`，**不准泛泛说"架构清晰"**。


## 4. 诉求 2：Workflow 工具场景穷举（仅深度档）

### 4.1 改动：`project-type-checklists.md` 的 f 类

将 **f. DevTool / Workflow 工具** 改为"通用追问 + 深度档专项"两层结构：

```
## f. DevTool / Workflow 工具

### 通用追问（各档通用，保留现有 4 条）
- 正常开发流程 / 产物文件 / 与 Git·CI·Spec·Issue 的关系 / 是否改代码

### 🔴 深度档专项：工作场景全景穷举
> 按「意图 → 入口 → 流程」三层展开，目标是覆盖"用户能拿它干哪些活"，一个不漏。

1. **用户意图 / 任务清单**：穷举用户能用它完成哪些任务，每个意图一行（看证据不看名字）。
2. **每个意图 → 对应入口**：哪些命令 / 触发方式服务这个意图？（带 `path`）
3. **每个入口 → 主路径 + 分支路径**：完整流程，含成功 / 失败 / 降级 / 特殊输入等关键分支。

**要求**：
- 覆盖**全部入口**，不漏。
- 一个意图可能对应多个入口，一个入口也可能服务多个意图——如实交叉标注。
- 每条结论带 `path:line`。
```

### 4.2 改动：`template-deep.md` 第 8 章

对 Workflow 类项目追加提示，并给一个"场景全景表"骨架：

```
### 8.x DevTool / Workflow 工具
- 判断依据：{{依赖 / 入口 / 目录证据}}

#### 工作场景全景（穷举）

| 用户意图 / 任务 | 对应入口 | 主路径 | 关键分支 | 源码位置 |
|---|---|---|---|---|
| {{意图1}} | {{命令/触发}} | {{主流程简述}} | {{失败/降级/特殊输入}} | `{{path:line}}` |
```

### 4.3 与第 5 章「端到端流程」的分工（重要，避免重复）

- **第 5 章端到端流程**：挑 1~2 条**最核心**流程，追到**最深、逐跳**（每一跳带 `path:line`、数据形态、副作用）。
- **第 8 章场景全景**：**广度覆盖**所有意图/入口，流程可略粗（主路径 + 关键分支即可），不必逐跳。
- 二者一深一广，互补不重复。

## 5. 诉求 3：Claude Code 扩展件解读 + 并行（仅深度档）

### 5.1 改动：`project-type-checklists.md` 新增 g 类

```
## g. Claude Code 插件 / AI 编程工具扩展

### 判断依据（看证据）
- `.claude/` 目录；`SKILL.md` 文件（skill）
- `commands/*.md` 或 `.claude/commands/`（slash command）
- `settings.json` / `settings.local.json` 里的 `hooks` 配置、`hooks/` 目录
- `agents/` 目录或 subagent 定义
- `plugin.json` / `.claude-plugin/`（插件清单）

### 逐条画像模板（每个 skill / command / hook / subagent 统一画像）
- **是什么 + 触发条件**：skill 的 description / command 的调用方式 / hook 的匹配时机与事件 / subagent 的角色定位
- **输入输出 / 副作用**：吃什么、产出什么、有无写文件/改代码/执行命令等副作用
- **内部关键逻辑**：带 `path:line`，引用了哪些 references / 脚本 / 模板
- **依赖与协作**：调用谁、被谁调用

### 整体装配图
> 这些扩展件如何"**技术编排**"成一个整体——哪个 command 触发哪个 skill、
> hook 在什么时机插入、subagent 被谁调起。**技术视角**，区别于诉求 2 的用户视角工作流。
```

### 5.2 改动：`parallel-strategy.md` 新增「按扩展条目并行」节

```
## 按扩展条目并行（服务诉求 3）

### 何时启用
- 深度档 + 命中 g 类（Claude Code 插件/扩展）+ 条目数较多（撑满上下文有风险）

### 切分方式
- **按单个条目切**：一个 skill / command / hook / subagent 各派一个子 agent。
- **分批**：一批不超过 5~6 个；在同一条消息里发出该批全部 Agent 调用才会真并行。

### 子 agent 产出
- 单条目**画像简报**（按 5.1 的逐条画像模板），中文、结构化、~400 字、必填"未搞清楚"。

### 主 agent 职责
- 聚合所有画像 → 统一术语、消除冲突。
- **自己画整体装配图**（装配图不外包——它需要全局视角）。

### 退化
- 当前环境无子 agent 工具时，退化为顺序解读，并在报告注明"未启用并行"。
```
